1. AUTO-ALLOCATION — New Backend Command + Mail
──────────────────────────────────────────

STATUS: NOT STARTED. No Console directory exists yet.

What to build:

  a) Migration (add 2 columns to distribution_plan_schedules):
     File: database/migrations/2026_06_04_000001_add_auto_allocation_to_distribution_plan_schedules.php
       + auto_allocated_at   TIMESTAMP NULL
       + auto_allocation_ref VARCHAR(100) NULL

  b) Artisan command:
     File: app/Console/Commands/AutoAllocateDistributionPlans.php
     Signature: schedule:auto-allocate-plans
     Logic:
       1. Query all plans WHERE status = 'planned' AND planned_date = today AND is_deleted = 0
       2. For each plan:
          a. buildInventoryCheckData($plan) — same method as controller
          b. IF can_proceed === true:
               - Call issuePlanItems() (same logic, reference = 'AUTO-...')
               - Set status = 'ready', auto_allocated_at = now(), auto_allocation_ref = reference
               - Send DistributionPlanAllocatedMail to admin(s)
          c. IF can_proceed === false:
               - Leave status = 'planned'
               - Send DistributionPlanShortfallMail to admin(s)
               - Email includes per-item shortage list with quantities

     NOTE: Extract issuePlanItems and buildInventoryCheckData into a shared
     DistributionPlanService so both the controller and the command can use them
     without duplicating the logic.

  c) Console Kernel:
     File: app/Console/Kernel.php
     Register: $schedule->command('schedule:auto-allocate-plans')->dailyAt('00:01');

  d) Mail classes:
     File: app/Mail/DistributionPlanAllocatedMail.php
       - Subject: "Auto-Allocated: [week_label] — [planned_date]"
       - Body: Plan name, template, target people, reference number, list of items issued
       - Recipients: all active admin users (role_id = admin/superadmin)

     File: app/Mail/DistributionPlanShortfallMail.php
       - Subject: "Stock Shortfall: [week_label] — [planned_date] — Action Required"
       - Body: Plan name, planned date, per-item shortfall table
                (item code | item name | required | available | shortage | location)
       - Recipients: all active admin users

  e) Service refactor (to avoid code duplication):
     File: app/Services/DistributionPlanService.php
     Methods to extract from controller:
       - buildInventoryCheckData(object $plan): array
       - issuePlanItems(object $plan, array $checkData, int $performedBy, ...): array
       - buildLocationBreakdown(object $plan, array $checkItems): array
       - resolveInventoryLocationId(): ?int
     Controller and Command both inject and call this service.

  f) Frontend — Add auto_allocated_at to ProgramPlanSummary interface
     File: batch-distribution.service.ts
     Add to ProgramPlanSummary:
       auto_allocated_at: string | null;
       auto_allocation_ref: string | null;

  g) Frontend — Show auto-allocation badge on plan card
     File: batch-distribution.component.html
     When plan.auto_allocated_at is set and plan.status === 'ready',
     show a small "Auto-allocated" chip next to the status badge.
     Tooltip: "Auto-allocated on [date] · Ref: [auto_allocation_ref]"

3. REMOVE STEP 2 FROM RUN BATCH MODAL
──────────────────────────────────────────

STATUS: NOT STARTED.

The execution modal (batch-distribution-execution.mixin.ts + HTML) currently has 3 steps:
  Step 1 — Stock Check
  Step 2 — Procure & Issue  (fill gaps, provide destination/reason)
  Step 3 — Complete

The reason step 2 has destination/reason fields is that the old flow required the user
to fill them in at run time. Now that we store them at schedule creation time, the
destination is the plan's week_label and the reason is optional (nullable).

Changes:

  a) BatchDistributionExecutionMixin:
     - When stock is sufficient (no shortages): skip step 2, go straight from step 1 → step 3
       i.e. canProceedFromStockCheck === true → proceedToCompleteStep() directly
     - When there ARE shortages: step 2 still needed for gap fill procured items,
       BUT remove the destination/reason inputs from step 2 — those come from the plan
     - issueStockForExecution() and confirmProcurementAndIssue() should pass:
         destination: plan.week_label
         reason: 'Scheduled Batch Distribution' (plan notes if present)
       instead of reading executionReason / executionDestination

  b) HTML — execution modal step 2 (lines ~620–773):
     - Remove the destination input
     - Remove the reason input (it's the required field causing the issue)
     - Remove canCompleteProcurementStep check that depends on executionReason
     - Keep only the gap-fill quantity inputs for shortage items

  c) HTML — execution modal step 1 (stock check step):
     - Change "Proceed to Fill Gaps" button (when no shortages):
       label → "Proceed →" and route straight to step 3 instead of step 2
     - Only show "Fill Gaps" (step 2) when there actually are shortages

  d) BatchDistributionExecutionMixin — update canCompleteProcurementStep:
     Remove the executionReason.trim() requirement:
     OLD: return !!this.executionReason.trim() && this.allGapsFilled && !this.executionIssuingStock;
     NEW: return this.allGapsFilled && !this.executionIssuingStock;

  e) ProgramPlanFinalCheckPayload in service:
     Make issue_reason optional (nullable) to match backend change:
     OLD: issue_reason: string;
     NEW: issue_reason?: string;
