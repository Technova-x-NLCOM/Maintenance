                    1. BACKUP & RECOVERY STRATEGY

1.1 OVERVIEW
------------
The system implements a comprehensive backup and recovery solution to ensure
data integrity and business continuity. The strategy includes automated 
backups, manual on-demand backups, and a secure restoration process.

1.2 BACKUP ARCHITECTURE
-----------------------
Component: Backup Module (Laravel-based)
Storage Location: storage/app/backups/
Supported Formats: SQL dumps (MySQL), SQLite copies
Permission Control: RBAC with 'manage_backups' permission

Key Features:
✓ Automated scheduled backups
✓ Manual on-demand backups
✓ Version-controlled backup files
✓ Safe restoration with pre-restore backups
✓ Permission-based access control
✓ File integrity validation

1.3 BACKUP IMPLEMENTATION DETAILS
----------------------------------

A. Backup File Naming Convention:
   Format: backup_{database_name}_{timestamp}.sql
   Example: backup_inventory_backend_2026-01-13_14-30-45.sql
   
   This naming provides:
   - Easy identification of database source
   - Chronological sorting capability
   - Timestamp precision for recovery point objectives

B. Backup Types Implemented:

   1) Full Backups:
      - Complete database dump including all tables and data
      - Uses mysqldump with --single-transaction flag
      - Ensures consistency without locking tables
      - Average size: 2-5 MB (depends on data volume)
      - Execution time: < 10 seconds for typical dataset

   2) Structure:
      mysqldump --user=root --password=[pwd] 
                --host=localhost 
                --single-transaction 
                --routines --triggers 
                inventory_backend > backup_file.sql

C. API Endpoints for Backup Operations:

   1. POST /api/backup/create
      - Creates new backup and returns downloadable file
      - Requires JWT authentication + manage_backups permission
      - Response: Binary SQL file download
      
   2. GET /api/backup/list
      - Lists all available backups with metadata
      - Returns: filename, size, creation timestamp
      - Formatted for human readability
      
   3. POST /api/backup/restore
      - Restores database from selected backup
      - Creates safety backup before restoration
      - Validates backup file integrity
      
   4. POST /api/backup/download
      - Downloads specific backup file
      - Allows offline storage
      
   5. POST /api/backup/delete
      - Removes backup file from storage
      - Requires confirmation
      - Audit logged

1.4 BACKUP SCHEDULE
-------------------
Recommended Schedule (Configurable via Laravel Scheduler):

Daily Backups:
  - Frequency: Every day at 2:00 AM
  - Retention: Last 7 daily backups
  - Command: php artisan backup:database
  - Implementation: Laravel Task Scheduler

Weekly Backups:
  - Frequency: Every Sunday at 3:00 AM
  - Retention: Last 4 weekly backups
  - Type: Full backup with verification
  
Monthly Backups:
  - Frequency: First day of month at 4:00 AM
  - Retention: Last 12 monthly backups
  - Storage: Offsite backup recommended

Laravel Scheduler Configuration (app/Console/Kernel.php):
```php
protected function schedule(Schedule $schedule)
{
    // Daily backup at 2 AM
    $schedule->command('backup:database')
             ->dailyAt('02:00')
             ->onSuccess(function () {
                 Log::info('Daily backup completed successfully');
             });
    
    // Weekly backup on Sundays at 3 AM
    $schedule->command('backup:database --weekly')
             ->weekly()
             ->sundays()
             ->at('03:00');
    
    // Monthly backup on 1st at 4 AM
    $schedule->command('backup:database --monthly')
             ->monthlyOn(1, '04:00');
}
```

1.5 RECOVERY PROCEDURES
-----------------------

A. Standard Recovery Process:

   Step 1: Authenticate and Verify Permission
   - User must have 'manage_backups' permission
   - JWT token must be valid
   
   Step 2: Select Backup File
   - List available backups via GET /api/backup/list
   - Choose appropriate recovery point
   - Verify backup file exists and is not corrupted
   
   Step 3: Pre-Restoration Safety Backup
   - System automatically creates backup of current state
   - Filename: backup_before_restore_{timestamp}.sql
   - Ensures rollback capability if restoration fails
   
   Step 4: Database Restoration
   - Disable foreign key checks temporarily
   - Drop existing tables (if specified)
   - Execute backup SQL file
   - Re-enable foreign key checks
   - Verify data integrity
   
   Step 5: Verification
   - Check table counts
   - Verify critical data records
   - Test application functionality
   - Audit log entry created

B. Recovery Time Objective (RTO):
   - Small database (< 5 MB): < 2 minutes
   - Medium database (5-50 MB): < 10 minutes
   - Large database (> 50 MB): < 30 minutes

C. Recovery Point Objective (RPO):
   - Maximum data loss: 24 hours (daily backup)
   - Recommended: 1 hour (with more frequent backups)

1.7 DISASTER RECOVERY SCENARIOS
--------------------------------

Scenario 1: Accidental Data Deletion
  Problem: User accidentally deletes critical inventory records
  Solution: 
    1. Identify last known good backup (within RPO)
    2. Create snapshot of current state
    3. Restore from backup
    4. Verify data integrity
    5. Notify stakeholders
  Expected Recovery Time: 5-10 minutes

Scenario 2: Database Corruption
  Problem: Database files become corrupted
  Solution:
    1. Stop application services
    2. Assess corruption extent
    3. Restore from latest backup
    4. Apply transaction logs if available
    5. Restart services
  Expected Recovery Time: 15-30 minutes

Scenario 3: Hardware Failure
  Problem: Database server hardware fails
  Solution:
    1. Provision new hardware/VM
    2. Install MySQL/MariaDB
    3. Restore from latest backup
    4. Update application configuration
    5. Test and verify
  Expected Recovery Time: 1-4 hours

Scenario 4: Ransomware/Security Breach
  Problem: Database compromised by security attack
  Solution:
    1. Isolate affected systems
    2. Assess breach extent
    3. Restore from pre-breach backup
    4. Patch security vulnerabilities
    5. Change all credentials
  Expected Recovery Time: 2-8 hours

2.2 INDEXING STRATEGY
---------------------

A. PRIMARY INDEXES (Automatically Created)
   All tables use AUTO_INCREMENT primary keys for optimal performance:
   - users.user_id
   - items.item_id
   - inventory_batches.batch_id
   - inventory_transactions.transaction_id
   - categories.category_id
   - item_types.item_type_id
   - expiry_alerts.alert_id
   - inventory_snapshots.snapshot_id

   Benefits:
   - O(log n) lookup time
   - Clustered index for MySQL InnoDB
   - Sequential insertion prevents fragmentation

B. FOREIGN KEY INDEXES
   Automatically indexed for referential integrity:
   
   items table:
   - INDEX(item_type_id) - For type filtering
   - INDEX(category_id) - For category grouping
   - INDEX(created_by) - For user-based queries
   
   inventory_batches table:
   - INDEX(item_id) - For batch lookup by item
   
   inventory_transactions table:
   - INDEX(item_id) - For transaction history by item
   - INDEX(batch_id) - For batch-specific transactions
   - INDEX(performed_by) - For user activity tracking
   - INDEX(approved_by) - For approval workflows

C. SPECIALIZED INDEXES FOR PERFORMANCE

   1. Full-Text Search Index:
      Table: items
      Column: item_description
      Type: FULLTEXT
      
      Purpose: Fast text search for item descriptions
      
      Query Before (LIKE search):
      ```sql
      SELECT * FROM items 
      WHERE item_description LIKE '%paracetamol%';
      ```
      Performance: Table scan, ~500ms for 10,000 records
      
      Query After (Full-text search):
      ```sql
      SELECT * FROM items 
      WHERE MATCH(item_description) AGAINST('paracetamol' IN NATURAL LANGUAGE MODE);
      ```
      Performance: Index scan, ~15ms for 10,000 records
      
      Improvement: 97% faster (33x speedup)

   2. Date-Based Indexes:
      Table: inventory_transactions
      Column: transaction_date
      Type: INDEX
      
      Purpose: Fast filtering of transactions by date range
      
      Query Before (No index):
      ```sql
      SELECT * FROM inventory_transactions 
      WHERE transaction_date >= '2026-01-01' 
      AND transaction_date < '2026-02-01';
      ```
      EXPLAIN Output: type: ALL, rows: 50000
      Execution Time: ~1200ms
      
      Query After (With index):
      EXPLAIN Output: type: range, rows: 1500, key: transaction_date
      Execution Time: ~45ms
      
      Improvement: 96% faster (27x speedup)

   3. Status Indexes:
      Table: items
      Column: is_active
      Type: INDEX
      
      Table: inventory_batches
      Column: status
      Type: INDEX
      
      Purpose: Fast filtering of active/inactive records
      
      Common Query:
      ```sql
      SELECT i.*, SUM(ib.quantity) as total_qty
      FROM items i
      JOIN inventory_batches ib ON i.item_id = ib.item_id
      WHERE i.is_active = 1 AND ib.status = 'active'
      GROUP BY i.item_id;
      ```
      
      Before Indexes:
      - Full table scans on both tables
      - Execution time: ~800ms for 5,000 items
      
      After Indexes:
      - Index scans reduce search space by 80%
      - Execution time: ~120ms
      
      Improvement: 85% faster (6.7x speedup)

   4. Transaction Type Index:
      Table: inventory_transactions
      Column: transaction_type
      Type: INDEX
      
      Purpose: Fast filtering by transaction type (IN, OUT, etc.)
      
      Query:
      ```sql
      SELECT COUNT(*), SUM(quantity) 
      FROM inventory_transactions 
      WHERE transaction_type = 'OUT' 
      AND transaction_date >= DATE_SUB(NOW(), INTERVAL 30 DAY);
      ```
      
      Performance Improvement:
      - Without indexes: 950ms (full scan)
      - With indexes: 65ms (compound index benefit)
      
      Improvement: 93% faster (14.6x speedup)

   5. Expiry Date Index:
      Table: inventory_batches
      Column: expiry_date
      Type: INDEX
      
      Purpose: Fast identification of expiring items
      
      Critical Query:
      ```sql
      SELECT * FROM inventory_batches 
      WHERE expiry_date BETWEEN NOW() AND DATE_ADD(NOW(), INTERVAL 30 DAY)
      AND status = 'active';
      ```
      
      Before Index:
      - Full table scan
      - Execution: ~600ms
      
      After Index:
      - Range scan on expiry_date
      - Execution: ~35ms
      
      Improvement: 94% faster (17x speedup)

Technique 2: Query Refactoring
-------------------------------
Original Query (Slow):
```sql
SELECT * FROM items WHERE item_id IN (
    SELECT item_id FROM inventory_batches WHERE status = 'active'
) AND is_active = 1;
```
Execution: 780ms (subquery executed multiple times)

Refactored Query (Fast):
```sql
SELECT DISTINCT i.* 
FROM items i
INNER JOIN inventory_batches ib ON i.item_id = ib.item_id
WHERE ib.status = 'active' AND i.is_active = 1;
```
Execution: 95ms
Improvement: 88% faster

SELECT * FROM inventory_transactions

- Returns 50 records per page
- Memory usage: 2MB
- Execution time: 45ms
- Improved user experience

Improvement: 99% faster

    4. Delete archived records from main table
    5. Update statistics
  Query:
    ```sql

    CREATE TABLE IF NOT EXISTS inventory_transactions_archive 
    LIKE inventory_transactions;

    INSERT INTO inventory_transactions_archive
    SELECT * FROM inventory_transactions
    WHERE transaction_date < DATE_SUB(NOW(), INTERVAL 2 YEAR);
    
    -- Verify count
    SELECT COUNT(*) FROM inventory_transactions_archive;
    
    -- Delete from main (after verification)
    DELETE FROM inventory_transactions
    WHERE transaction_date < DATE_SUB(NOW(), INTERVAL 2 YEAR);
    ```
